# 03_query — Query & Semantic Access (Layer 4)

Versioned Cypher queries that answer architecture questions against the Neo4j graph. This is the primary interface between the graph data and the generation layer above it.

> The MCP `archim8_run_cypher_query` and `archim8_read_architecture_view` tools draw directly from this library at runtime.

---

## Structure

```
03_query/
└── cypher/
    ├── config/                        # Query runner config and parameters
    ├── library/                       # Versioned query files by concern
    │   ├── callpaths/                 # Bounded call chains (placeholder)
    │   ├── entrypoints/               # Main methods, controllers, entry points
    │   ├── jdeps/                     # jdeps-specific: ingest, export, sampling
    │   ├── jqa/                       # jQAssistant structural queries
    │   ├── modules/                   # Package and Maven module analysis
    │   ├── quality/                   # Graph integrity checks (placeholder)
    │   └── runtime_primitives/        # IO / date / numeric usage (placeholder)
    └── scripts/                       # Query execution helpers
```

---

## Query library

### `jqa/` — structural analysis queries

| File | What it returns |
|------|----------------|
| `violations.cypher` | Layering violations detected by jQA rules |
| `module-deps.cypher` | Module-to-module dependency matrix (Maven level) |
| `grpc-services.cypher` | gRPC service definitions and their methods |
| `spring-components.cypher` | Spring-annotated types and their roles |
| `observability-coverage.cypher` | Types with/without observability annotations |
| `pekko-http-api.cypher` | Pekko HTTP route definitions |
| `key-abstractions.cypher` | High-connectivity types (interfaces, base classes) |
| `cobol-subsystem.cypher` | COBOL-emulation layer analysis |

### `entrypoints/`

| File | What it returns |
|------|----------------|
| `find_entrypoints.cypher` | All application entry points (main, controller, handler types) |

### `modules/`

| File | What it returns |
|------|----------------|
| `all_modules.cypher` | All Maven modules with package counts |

### `jdeps/`

| File | Purpose |
|------|---------|
| `neo4j-ingest-jdeps.cypher` | LOAD CSV — imports `jdeps-jar-edges.csv` into Neo4j |
| `neo4j-export-jdeps.cypher` | APOC export — writes jar-deps, top-incoming, top-outgoing, cycles CSVs |
| `neo4j-export-sample.cypher` | Sample export query for validation |

### Placeholder sub-folders

These exist for future queries. Add `.cypher` files and they are picked up automatically by `make generate-views`:

- `callpaths/` — bounded call chains
- `quality/` — graph integrity and consistency checks
- `runtime_primitives/` — IO, date, numeric API usage

## How to run

Views are generated by the generate layer — not run directly:

```powershell
make generate-views          # Run all library queries -> 05_deliver/output/views/*.md
make generate-views FORCE=1  # Regenerate even views already in manifest
make generate-views VIEW=<name>  # Regenerate a single named view
```

Individual Cypher files can also be run directly against Neo4j:

```powershell
# Via cypher-shell in the Neo4j container
docker exec archim8-neo4j cypher-shell -u neo4j -p password --file /path/to/query.cypher

# Or paste into Neo4j Browser at http://localhost:7474
```

---

## Adding a new query

1. Create a `.cypher` file in the appropriate sub-folder (or a new one)
2. Write your query with a `RETURN` clause producing a result set
3. Run `make generate-views` — the manifest is updated automatically
4. The MCP tools discover it via `archim8_list_available_views` on the next run

There is no registration step — the generator script discovers all `.cypher` files recursively.
